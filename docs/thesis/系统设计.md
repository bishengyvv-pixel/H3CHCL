# 系统设计

> 对应论文章节：第4章 系统设计

---

## 1. 设计目标

本项目面向 H3C 园区网络的安全自动化评估，在设计上追求以下目标：

| 目标 | 实现策略 |
|------|----------|
| **分层解耦** | 6 个功能层 + 1 个模型层，层间仅通过 dataclass 传递数据 |
| **配置驱动** | 设备清单、规则库、运行参数全部 YAML 化，新增检测项零代码改动 |
| **容错优先** | 单设备失败不中断全局流程，异常全量捕获并记录 |
| **可观测性** | 四阶段流水线，每阶段输出日志，辅以摘要和明细 |

## 2. 分层架构设计

### 2.1 分层示意

```
┌──────────────────────────────────────────────────────────────┐
│                    CLI 层 (main.py)                           │
│              AssessmentPipeline 流水线编排                     │
├──────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌──────────────┐  ┌──────────────────┐    │
│  │ config/     │  │ connector/   │  │ engine/          │    │
│  │ 配置加载    │  │ SSH连接管理   │  │ 并发调度+限速     │    │
│  └─────────────┘  └──────────────┘  └──────────────────┘    │
│  ┌─────────────┐  ┌────────────────────────────────────┐    │
│  │ identifier/ │  │ evaluator/                         │    │
│  │ 角色识别    │  │ 规则匹配 + 风险计分                  │    │
│  └─────────────┘  └────────────────────────────────────┘    │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ reporter/                                            │   │
│  │ Excel报告 + JSON报告 + PDF预留                        │   │
│  └──────────────────────────────────────────────────────┘   │
├──────────────────────────────────────────────────────────────┤
│  ┌──────────────────────────────────────────────────────┐   │
│  │ models/  (数据模型层)                                  │   │
│  │ Device · Rule · AssessmentResult · FailedItem         │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 层间依赖关系

```
main.py (CLI)
  ├── config/loader.py ──→ models/Device, Rule, Settings
  ├── connector/ ────────→ models/Device
  ├── engine/ ───────────→ 无模型依赖（纯基础设施）
  ├── identifier/ ───────→ models/Device
  ├── evaluator/ ────────→ models/Device, Rule, AssessmentResult
  └── reporter/ ─────────→ models/AssessmentResult
```

**依赖规则**：
- 上层可依赖下层，下层不依赖上层
- 所有层间数据传递使用 `models/` 中的 dataclass
- 各层内部修改不对外暴露实现细节

## 3. 模块详细设计

### 3.1 配置加载模块 (config/)

**类**：`ConfigLoader`

```
ConfigLoader
  ├── load_devices()  → list[Device]
  ├── load_rules()    → list[Rule]
  └── load_settings() → Settings
```

**设计要点**：
- YAML 文件缺失时返回空列表/默认值，不抛异常
- Device 构造时 `source_ip` 可选字段，未填则为空字符串
- Rule 构造时 `match_type` 和 `applicable_roles` 提供合理的默认值

### 3.2 设备连接模块 (connector/)

**类**：`DeviceConnection`

```
DeviceConnection(device, device_type, timeout, global_source_ip)
  ├── connect()           # 建立SSH → 获取hostname → 设置status
  │     ├── socket.create_connection(source_address=(source_ip, 0))
  │     └── ConnectHandler(sock=sock, ...)
  ├── execute_commands()  # {cmd: output, ...}
  └── disconnect()

retry_on_failure(max_attempts=3, delay=1.0)   # 装饰器
```

**源IP绑定流程**：

```
1. 确定 source_ip：device.source_ip or global_source_ip
2. 创建 socket，绑定到 source_ip:随机端口
3. 连接到 target_ip:22
4. 将预绑定的 socket 传入 Netmiko ConnectHandler
```

### 3.3 并发调度模块 (engine/)

**类**：`TaskScheduler`、`RateLimiter`

```
TaskScheduler(max_workers, rate_limiter)
  └── run_all(tasks: list[Callable]) → list[result]
        ├── ThreadPoolExecutor(max_workers)
        ├── as_completed() 收集结果
        ├── 异常不中断，记录日志
        └── 每个 future 完成 → rate_limiter.tick()

RateLimiter(batch_delay)
  └── tick()  # time.sleep(batch_delay)
```

**设计理由**：
- `as_completed` 而非 `wait`：先完成的先处理，I/O 密集场景更高效
- `batch_delay` 的必要性：HCL 虚拟设备运行在 VirtualBox 上，无休眠的并发 SSH 可能导致宿主机 CPU 100%

### 3.4 角色识别模块 (identifier/) — **核心算法**

**类**：`RoleIdentifier`

**算法流程**：

```
输入: Device.collected_outputs

步骤1: 设备类型判定
  ├── 读取 "display version" 输出
  ├── 正则匹配路由器型号: \b(MSR\d+|SR\d+|CR\d+|RT\d+|Router|AR\d+)
  └── 命中 → ROUTER (return)

步骤2: 特征提取 ("display current-configuration")
  ├── access_port_cnt = len(re.findall(r"port\s+link-type\s+access"))
  ├── vlan_if_cnt     = len(re.findall(r"^interface\s+Vlan-interface", re.M))
  ├── has_ospf  = re.search(r"ospf\s+\d+")
  ├── has_bgp   = re.search(r"bgp\s+\d+")
  ├── has_vrrp  = re.search(r"vrrp\s+vrid")
  ├── has_edged = re.search(r"stp\s+edged-port")
  └── has_dhcp  = re.search(r"dhcp-snooping")

步骤3: 评分计算
  ├── L3 = bgp*4 + ospf*2 + vrrp*2 + (vlan>=5)*3 + (vlan>=2)*1
  └── L2 = (access>=10)*3 + (access>=5)*2 + edged*2 + dhcp*2

步骤4: 阈值判定
  ├── L3 ≥ 5                   → CORE
  ├── L3 ≥ 2  且 L2 == 0       → AGGREGATION
  ├── L3 ≥ 2  且 L2 ≥ 1        → ACCESS
  ├── L2 ≥ 1                   → ACCESS
  └── 其他                     → UNKNOWN
```

**设计理由**：
- 不单纯依赖 OSPF (在 HCL 实验环境中即使接入层设备也可能启用 OSPF)
- L2 特征（edged-port）用于区分"纯L3汇聚"和"三层接入交换机"
- 路由器型号匹配优先于交换机评分，避免路由器被误判

### 3.5 规则评估模块 (evaluator/)

**类**：`RuleEngine`、`RiskCalculator`

```
RuleEngine.evaluate(device, rules)
  ├── 过滤规则: applicable_roles 为空 || device.role in applicable_roles
  ├── 取 device.collected_outputs[rule.check_cmd]
  ├── re.search(rule.regex, output)
  └── 匹配判定:
        ├── match_type="find"     + 未匹配 → FailedItem (扣分)
        └── match_type="not_find" + 已匹配 → FailedItem (扣分)

RiskCalculator.calculate(device, failed_items)
  ├── score = 100 - sum(item.weight for item in failed_items)
  ├── score = max(score, 0)
  ├── is_fatal = any(item.rule_id ∈ FATAL_RULE_IDS)
  ├── if score < 60 or is_fatal → high
  ├── elif score < 85 → medium
  └── else → low
```

**match_type 设计**：

| 规则类型 | match_type | 示例 |
|----------|-----------|------|
| "该有的安全配置没找到" | `find` | `info-center enable` — 未找到说明日志未开启 |
| "不该有的风险配置被找到" | `not_find` | 弱口令 regex — 找到说明存在弱密码 |

## 4. 安全规则库设计

### 4.1 规则分层体系

```
通用规则 (6条)
  ├── 弱口令检测 (20分)
  ├── 日志审计   (10分)
  ├── NTP同步    (5分)
  ├── SSH服务    (10分)
  ├── 密码强度   (5分)
  └── 空闲超时   (5分)
      │
      ├── 路由器专属 (3条)    总分35
      │   ├── Console认证
      │   ├── VTY ACL
      │   └── HTTP关闭
      │
      ├── 接入层专属 (5条)    总分60
      │   ├── BPDU保护
      │   ├── DHCP Snooping
      │   ├── 环路检测
      │   ├── 风暴控制
      │   └── 端口安全
      │
      ├── 汇聚层专属 (3条)    总分35 + 40(接入通用) = 75
      │   ├── 管理ACL
      │   ├── VRRP认证
      │   └── VTY ACL
      │
      └── 核心层专属 (3条)    总分35 + 40(接入通用) = 75
          ├── 路由认证
          ├── SNMP ACL
          └── 控制平面限速
```

### 4.2 设计原则

1. **角色责任分离**：接入层关注二层安全（STP/DHCP），核心层关注三层安全（路由/SNMP/控制平面）
2. **权重对应风险**：致命项（弱口令20分）> 严重项（BPDU/DHCP/ACL 15分）> 一般项（NTP/空闲超时 5分）
3. **正负向双模式**：`find` 模式匹配合规特征，`not_find` 模式匹配风险特征

## 5. 数据流设计

### 5.1 完整数据流

```
1. ConfigLoader 加载 YAML → Device[] + Rule[] + Settings
2. TaskScheduler 使用 ThreadPoolExecutor 提交 DeviceConnection 任务
3. DeviceConnection.connect → SSH → 采集 7 条 display 命令 → Device.collected_outputs
4. RoleIdentifier.identify → 型号匹配/特征评分 → Device.role
5. RuleEngine.evaluate → 规则过滤 → Regex匹配 → FailedItem[]
6. RiskCalculator.calculate → 加权扣分 → AssessmentResult
7. ExcelReporter / JsonReporter → 报告文件
```

### 5.2 异常处理流

```
单台设备连接失败
  → DeviceConnection.connect() 返回 False
  → Device.status = ERROR, Device.error_message = 错误信息
  → Device.collected_outputs = {} (空字典)
  → 后续阶段自动跳过（online = [d for d if d.collected_outputs != {}]）
  → 日志记录: "✗ 172.16.255.1 — TCP connection failed"
  → 不影响其他设备
```
